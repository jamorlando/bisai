# 实训成果智能核查与评价系统软件功能需求分析文档

## 一、引言

### 1.1 编写目的

本文档用于明确“实训成果智能核查与评价系统”的软件功能需求、用户角色、业务流程、数据需求、接口需求和非功能需求，为后续系统设计、编码实现、测试验收和比赛答辩提供依据。

### 1.2 项目背景

本系统面向实训教学成果管理场景，解决教师人工检查实训报告、截图、代码包、测试结果和总结材料工作量大、评价标准不统一、反馈效率低等问题。系统通过文档解析、OCR、多模态大模型、RAG 检索增强、规则评分和教师复核机制，实现实训成果的智能核查、辅助评分和报告生成。

系统需要适配 LoongArch 与麒麟高级服务器操作系统环境，并支持在模型服务不可用时降级为人工评分模式。

### 1.3 适用范围

本文档适用于以下人员：

- 项目开发人员
- 测试人员
- 指导教师
- 比赛评审人员
- 后续维护人员

### 1.4 术语表

| 术语 | 说明 |
|---|---|
| MVP | 最小可行版本，指优先实现可完成核心业务闭环的功能集合 |
| OCR | Optical Character Recognition，文字识别技术，用于从图片、扫描件中提取文字 |
| RAG | Retrieval-Augmented Generation，检索增强生成，用于将知识库内容检索后提供给大模型参考 |
| LazyGraphRAG | 轻量级图检索增强思路，用于复杂关系、多跳关联场景的检索增强 |
| LoongArch | 龙架构 CPU 指令集架构，属于国产 CPU 架构之一 |
| 麒麟高级服务器操作系统 | 国产服务器操作系统，作为本系统目标部署环境之一 |
| Caffeine | Java 本地缓存组件，MVP 阶段用于减少外部缓存依赖 |
| Redis | 分布式缓存和消息队列组件，可用于缓存、限流和异步任务 |
| pgvector | PostgreSQL 的向量扩展，用于存储和检索 Embedding 向量 |
| Embedding | 将文本转换为向量表示的模型能力，用于相似度检索 |
| Rerank | 对向量召回结果进行二次排序，提高检索相关性 |
| 多模态大模型 | 能处理文本、图片、截图等多种输入的大模型 |
| JWT | JSON Web Token，用于前后端分离场景下的登录认证和身份传递 |

## 二、总体需求

### 2.1 系统目标

系统建设目标如下：

1. 支持学生在线提交实训成果文件。
2. 支持教师创建课程、实训任务和评价指标。
3. 自动解析 Word、PDF、图片、Excel、ZIP 等成果文件。
4. 自动核查学生成果是否满足实训任务要求。
5. 自动识别成果中的缺失项、逻辑漏洞和高风险问题。
6. 根据评分标准生成系统建议分和评分理由。
7. 支持教师复核、调整分数并发布最终成绩。
8. 支持个人评价报告和班级统计报表导出。
9. 支持系统配置、任务监控、日志审计和安全控制。
10. 支持国产化部署和 AI 能力降级策略。

### 2.2 用户角色

系统包含三类主要用户：

| 角色 | 说明 |
|---|---|
| 学生 | 查看实训任务、上传成果文件、查看评价结果和下载个人报告 |
| 教师 | 创建课程和任务、配置评分标准、启动智能评价、复核成绩和导出报表 |
| 管理员 | 管理用户、班级、课程、模型配置、系统参数、日志和数据备份 |

### 2.3 运行环境需求

系统采用 B/S 架构，用户通过浏览器访问。

基础运行环境：

- CPU 架构：LoongArch
- 操作系统：麒麟高级服务器操作系统
- 后端：Spring Boot
- 前端：Vue 3 + TypeScript + Element Plus
- 数据库：PostgreSQL / openGauss / 达梦数据库
- 文件存储：MinIO 或本地文件系统
- AI 能力：ModelScope API、本地模型服务或旁路模型服务

## 三、功能需求

### 3.1 登录与权限管理

#### 3.1.1 用户登录

系统应支持学生、教师、管理员通过账号密码登录。

功能要求：

- 支持用户名和密码登录。
- 支持登录验证码。
- 支持登录失败次数限制。
- 支持首次登录修改初始密码。
- 支持登录状态保持和退出登录。

#### 3.1.2 权限控制

系统采用基于角色的权限控制。

功能要求：

- 学生只能查看和操作自己的任务、提交记录和评价结果。
- 教师只能管理自己课程和任务下的学生成果。
- 管理员可管理系统基础数据和运行配置。
- 所有关键操作需要记录操作日志。

### 3.2 基础数据管理

#### 3.2.1 用户管理

管理员应能够维护系统用户。

功能要求：

- 新增、编辑、禁用、启用用户。
- 设置用户角色。
- 维护学生所属班级。
- 维护教师基本信息。
- 重置用户密码。

#### 3.2.2 班级管理

管理员应能够维护班级信息。

功能要求：

- 新增班级。
- 编辑班级名称、年级、专业等信息。
- 查看班级学生列表。
- 禁用或启用班级。

#### 3.2.3 课程管理

教师或管理员应能够维护课程信息。

功能要求：

- 创建课程。
- 绑定授课教师和班级。
- 编辑课程说明。
- 查看课程下的实训任务。
- 启用或停用课程。

### 3.3 实训任务管理

教师应能够创建和发布实训任务。

功能要求：

- 创建实训任务。
- 设置任务名称、任务说明和实训要求。
- 设置开始时间和截止时间。
- 设置允许上传的文件类型和大小。
- 设置是否允许重新提交。
- 选择评价模板。
- 发布、编辑、关闭任务。
- 查看任务提交情况。

业务规则：

- 已发布任务允许学生查看和提交。
- 超过截止时间后，系统默认不允许提交，除非教师开启补交。
- 同一任务应绑定固定评价模板，保证评分一致性。

### 3.4 成果提交管理

#### 3.4.1 学生上传成果

学生应能够按任务要求上传实训成果。

支持文件类型：

- Word：`.doc`、`.docx`
- PDF：`.pdf`
- 图片：`.jpg`、`.jpeg`、`.png`
- Excel：`.xls`、`.xlsx`
- 压缩包：`.zip`

功能要求：

- 支持单文件和多文件上传。
- 支持文件大小和类型校验。
- 支持重新提交并保留历史版本。
- 支持上传进度提示。
- 支持查看提交状态。

版本管理规则：

- 每次重新提交生成一个新的提交版本号。
- MVP 阶段默认保留最近 5 个历史版本，管理员可通过系统配置调整保留数量。
- 最新有效版本指“截止时间前最后一次提交且文件校验通过的版本”。如果该版本解析失败，系统仍以该版本作为当前版本，但评分页面应提示解析失败并允许教师选择人工评分或回退参考上一个解析成功版本。
- 教师开启补交后，评分基于教师确认的补交版本。
- 学生重新提交后，旧版本解析结果和评分结果保留为历史记录，不再作为当前评分依据。
- 新版本提交后，系统应重新触发文档解析、OCR、向量化和智能核查任务。
- 已发布成绩后，学生不得自行重新提交；如需更换文件，必须由教师退回或重新开放提交。

#### 3.4.2 文件在线预览

教师复核时应能够在线查看学生原始成果。

功能要求：

- PDF 文件使用 pdf.js 在线预览。
- Word 文件可转换为 PDF 后预览。
- 图片文件支持在线查看和缩放。
- Excel 文件支持表格预览或 PDF 预览。
- ZIP 文件展示安全解压后的文件目录。

安全要求：

- 预览接口必须进行权限校验。
- ZIP 解压需限制总大小、文件数量和嵌套层数。
- 禁止路径穿越文件。

### 3.5 文档解析功能

系统应自动解析学生上传的成果文件。

功能要求：

- 解析 Word 文档中的标题、正文、表格和图片说明。
- 解析 PDF 文档中的文本、页码和图文内容。
- 对图片和扫描件进行 OCR 识别。
- 对复杂截图和图表调用多模态模型分析。
- 对 Excel 文件提取表格数据和统计结果。
- 对 ZIP 文件进行安全解压并识别其中的文档、图片和代码文件。
- 将解析结果保存为结构化文本和 JSON 数据。

处理策略：

- 优先使用文档解析和 OCR 获取可检索文本。
- 对 OCR 置信度低或图文关系复杂的内容调用多模态模型。
- 解析失败时记录失败原因，并允许教师人工查看文件。

### 3.6 知识库管理

系统应支持维护课程知识库，为 RAG 检索和评分提供参考依据。

功能要求：

- 创建课程知识库。
- 上传实训指导书、评分标准、课程知识点和历史优秀样例。
- 对知识库文档进行解析、切片和向量化。
- 设置知识库适用课程和任务范围。
- 启用或停用知识库文档。
- 查看知识库文档解析状态和向量化状态。

业务规则：

- 知识库内容仅作为模型评分参考。
- 最终评分以任务评价模板和教师确认结果为准。
- 系统优先检索当前任务绑定的评分标准，其次检索课程知识点和历史优秀样例。
- 检索结果只影响大模型评分建议和评分理由，不直接修改客观规则分。
- 检索不到相关内容时，只影响主观评分参考资料，不影响客观评分；系统应基于任务评价模板和学生成果内容继续评分，并在评分理由中标注“未检索到相关知识库依据”。
- 检索结果与学生成果无关时，不得强行引用为评分依据。
- 知识库文档更新后，不自动改变已发布成绩；如教师需要使用新知识库重新评价，应手动发起重评并记录原因。

### 3.7 智能核查功能

#### 3.7.1 实训要求匹配

系统应根据教师配置的实训要求，自动检查学生成果是否满足要求。

核查粒度为“任务要求是否被覆盖”，重点判断教师明确要求的成果项是否出现。

核查内容包括：

- 是否包含需求分析。
- 是否包含系统设计。
- 是否包含数据库设计。
- 是否包含功能实现说明。
- 是否包含运行截图。
- 是否包含测试用例和测试结果。
- 是否包含实训总结。
- 是否提交源代码或项目包。

输出结果：

- 已完成。
- 部分完成。
- 未完成。
- 问题说明。
- 证据片段。
- 修改建议。

#### 3.7.2 实训步骤完整性核查

系统应检查学生成果是否覆盖完整实训流程。

核查粒度为“实训过程是否连贯完整”，重点判断成果从任务理解到总结反思的过程链路是否闭合。

流程模板默认包括：

```text
任务理解 -> 需求分析 -> 系统设计 -> 功能实现 -> 测试验证 -> 总结反思
```

该流程不应硬编码，教师可在实训任务或评价模板中调整步骤名称、顺序和必填要求。

核查内容：

- 是否缺少关键步骤。
- 是否存在章节标题但正文为空。
- 是否存在内容顺序混乱。
- 是否缺少关键图表。
- 是否缺少运行结果证明。
- 是否缺少测试验证过程。

#### 3.7.3 逻辑漏洞识别

系统应结合规则引擎和大模型识别成果中的逻辑问题。

识别内容：

- 需求与实现不一致。
- 数据库设计无法支撑功能描述。
- 功能截图与文字说明不匹配。
- 测试用例未覆盖核心功能。
- 系统架构说明前后矛盾。
- 实验结果与总结结论不一致。
- 报告内容过于空泛或存在复制粘贴痕迹。

### 3.8 智能评分功能

#### 3.8.1 评价指标配置

教师应能够配置评价模板和评价指标。

功能要求：

- 创建评价模板。
- 设置一级指标和二级指标。
- 设置指标权重。
- 设置单项满分。
- 设置评分规则。
- 绑定评价模板到实训任务。

#### 3.8.2 客观评分

系统应根据明确规则自动计算客观分。

评分依据：

- 是否按时提交。
- 文件格式是否符合要求。
- 是否包含必要章节。
- 是否包含运行截图。
- 是否包含测试结果。
- 字数是否达标。

#### 3.8.3 主观评分建议

系统应调用大模型生成主观评分建议。

评价内容：

- 需求分析质量。
- 设计方案合理性。
- 功能实现完整度。
- 测试验证充分性。
- 文档表达清晰度。
- 总结反思深度。
- 创新性和应用价值。

输出内容：

- 各指标建议分。
- 评分理由。
- 证据片段。
- 风险提示。
- 改进建议。

#### 3.8.4 教师复核

教师应能够查看系统评分建议并确认最终成绩。

功能要求：

- 查看系统建议分。
- 查看客观评分命中项。
- 查看主观评分理由。
- 查看风险扣分依据。
- 修改单项分数。
- 填写教师评语。
- 发布最终成绩。

业务规则：

- 系统评分仅作为辅助。
- 最终成绩必须由教师确认。
- 教师修改分数需要记录操作日志。

异常处理规则：

- 大模型返回结果不是合法 JSON 时，系统自动重试一次；仍失败则标记为模型输出异常。
- 大模型调用超时时，任务进入失败状态并记录错误原因，教师可手动重试。
- 模型返回分数超过指标满分时，系统按指标满分截断，并记录异常。
- 模型返回负分时，系统按 0 分处理，并记录异常。
- 模型返回字段缺失时，系统保留可解析字段，缺失字段提示教师人工补充。
- 智能评分失败不影响教师进入人工评分流程。

### 3.9 通知与消息功能

系统应提供站内消息机制，用于提醒关键业务状态变化。

通知场景：

- 学生提交成果后，教师收到待评价提醒。
- 学生重新提交成果后，教师收到版本更新提醒。
- 智能评价完成后，教师收到待复核提醒。
- 教师发布成绩后，学生收到评价结果提醒。
- 批量任务失败时，教师或管理员收到失败提醒。
- 模型调用配额接近上限时，管理员收到预警提醒。

功能要求：

- 支持站内消息列表。
- 支持消息已读和未读状态。
- 支持按任务、课程、消息类型筛选。
- MVP 阶段优先实现站内消息，短信、邮件和企业微信通知作为可选扩展。

### 3.10 报表生成与导出

#### 3.10.1 学生个人评价报告

系统应支持生成学生个人评价报告。

报告内容：

- 学生基本信息。
- 实训任务信息。
- 提交文件信息。
- 成果摘要。
- 完整性核查结果。
- 逻辑问题提示。
- 各指标得分。
- 总分。
- 教师评语。
- 改进建议。

导出格式：

- Word
- PDF

#### 3.10.2 班级统计报表

系统应支持生成班级统计报表。

统计内容：

- 班级平均分。
- 最高分和最低分。
- 分数段分布。
- 各指标平均得分。
- 未提交学生名单。
- 高风险成果名单。
- 常见问题统计。

导出格式：

- Excel
- PDF

### 3.11 批量任务与进度管理

教师应能够批量启动智能评价和批量导出报告。

功能要求：

- 批量启动文档解析。
- 批量启动智能核查。
- 批量生成评分建议。
- 批量导出个人报告。
- 批量导出班级报表。
- 查看任务总数、成功数、失败数和执行中数量。
- 查看失败原因并支持重新执行。

并发控制：

- 同一提交记录不能重复进入同类任务队列。
- 大模型 API 达到限流或配额上限时暂停队列。
- 失败任务可单独重试，不影响其他任务。

### 3.12 系统配置与日志管理

管理员应能够维护系统运行配置。

配置内容：

- 模型 API 地址。
- 模型密钥引用。
- OCR 服务地址。
- 调用超时时间。
- 每日调用配额。
- 文件上传大小限制。
- 缓存策略。

日志内容：

- 用户登录日志。
- 文件上传日志。
- 模型调用日志。
- 评分修改日志。
- 系统异常日志。
- 操作审计日志。

## 四、业务流程需求

### 4.1 学生提交流程

```text
学生登录
  ↓
查看实训任务
  ↓
上传成果文件
  ↓
系统校验文件
  ↓
保存文件和提交记录
  ↓
进入待解析状态
```

### 4.2 智能评价流程

```text
读取提交文件
  ↓
文档解析 / OCR / 多模态理解
  ↓
文档切片与向量检索
  ↓
匹配实训要求和评分标准
  ↓
完整性核查
  ↓
逻辑漏洞识别
  ↓
生成评分建议
  ↓
生成初评报告
```

### 4.3 教师复核流程

```text
教师查看系统评分
  ↓
查看核查问题和证据
  ↓
在线预览原始文件
  ↓
调整单项分数
  ↓
填写教师评语
  ↓
确认最终成绩
  ↓
发布评价结果
```

### 4.4 人工降级流程

当模型 API 不可用、本地模型无法部署或网络中断时，系统进入人工评分模式。

```text
教师查看学生提交文件
  ↓
查看基础解析结果
  ↓
按评价模板逐项打分
  ↓
填写评语
  ↓
确认最终成绩
  ↓
导出评价报告
```

### 4.5 评分校准流程

为保证同一任务下不同学生成果的评分尺度一致，系统应提供基础评分校准流程。

```text
教师选择实训任务
  ↓
选择评价模板
  ↓
抽取 2-3 份典型成果作为校准样本
  ↓
教师人工确认样本分数和评分理由
  ↓
系统记录校准参考
  ↓
后续智能评分参考同一评分尺度
```

MVP 阶段至少要求同一任务绑定固定评价模板、固定评分规则和固定 Prompt 版本。教师批量评价前可先对少量样本进行人工复核，确认评分尺度后再批量处理。

校准数据包括样本提交 ID、指标得分、教师评分理由、典型优点、典型问题和扣分依据。MVP 阶段校准结果通过 Prompt 上下文提供给大模型，作为同一任务后续评分的参考样例；系统不自动对模型分数做线性映射。正式评分结果仍需经过分数边界校验和教师确认。

### 4.6 异常业务流程

#### 4.6.1 学生误提交处理

- 截止时间前，学生可重新提交，系统以最新有效版本作为当前版本。
- 截止时间后，学生不能自行撤回或覆盖提交。
- 如学生误提交，需联系教师退回或开启补交。
- 教师退回后，系统记录退回原因，并通知学生重新提交。

#### 4.6.2 成绩发布后修改

- 教师发布成绩后，学生可查看结果和下载报告。
- 如发现评分错误，教师可发起成绩修正。
- 成绩修正必须填写原因，并记录操作日志。
- 修正后系统重新生成评价报告，并通知学生查看最新结果。

#### 4.6.3 任务关闭后处理

- 任务关闭后，学生不能继续提交或修改成果。
- 教师仍可查看、导出和复核已提交数据。
- 管理员不得直接删除已评分数据，除非执行归档或数据清理流程。
- 任务关闭不影响历史报告下载。

#### 4.6.4 文件解析失败处理

- 系统应记录失败文件、失败阶段和错误原因。
- 教师可选择重新解析、下载原文件或进入人工评分。
- 如果单个文件解析失败，不应阻塞同一提交记录中其他文件的解析。

### 4.7 业务规则汇总

| 规则类别 | 规则说明 |
|---|---|
| 提交规则 | 截止时间前允许学生重新提交，截止时间后需教师开启补交 |
| 版本规则 | 评分默认基于最新有效版本，历史版本保留但不参与当前评分 |
| 任务规则 | 已发布任务学生可见，已关闭任务学生不可继续提交 |
| 评分规则 | 系统评分仅作为建议，最终成绩必须由教师确认 |
| 校准规则 | 同一任务使用固定评价模板、固定评分规则和固定 Prompt 版本 |
| 知识库规则 | 知识库只影响主观评分建议和评分理由，不直接修改客观规则分 |
| 发布规则 | 成绩发布后如需修改，必须走成绩修正流程并记录原因 |
| 文件规则 | 文件访问必须校验权限，删除优先采用逻辑删除 |
| 降级规则 | AI 服务不可用时，系统保留人工评分和报表导出能力 |

## 六、接口需求

### 6.1 前后端接口规范

系统前后端采用 RESTful API 进行通信，数据格式统一使用 JSON。

接口规范：

- 请求路径使用 `/api` 作为统一前缀。
- 使用 HTTP Method 表达操作语义：`GET` 查询、`POST` 新增、`PUT` 修改、`DELETE` 删除。
- 登录后接口需携带认证信息，采用 JWT 或 Session。
- 文件上传接口使用 `multipart/form-data`。
- 普通业务接口返回统一响应结构。
- 列表查询接口统一支持分页和排序。

统一响应格式：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

异常响应格式：

```json
{
  "code": 40001,
  "message": "参数错误",
  "data": null
}
```

分页请求参数：

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| page | INTEGER | 1 | 页码，从 1 开始 |
| size | INTEGER | 20 | 每页条数，最大不超过 100 |
| sort | STRING | created_at | 排序字段 |
| order | STRING | desc | 排序方向，asc 或 desc |

分页响应格式：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "items": [],
    "page": 1,
    "size": 20,
    "total": 100
  }
}
```

错误码规范：

| 错误码范围 | 类型 | 示例 |
|---|---|---|
| 0 | 成功 | 0 表示成功 |
| 40000-40999 | 请求参数错误 | 40001 参数错误，40002 文件类型不支持 |
| 40100-40199 | 认证错误 | 40101 未登录，40102 登录已过期 |
| 40300-40399 | 权限错误 | 40301 权限不足，40302 数据越权 |
| 40400-40499 | 资源不存在 | 40401 任务不存在，40402 文件不存在 |
| 40900-40999 | 业务冲突 | 40901 重复提交，40902 状态不允许操作 |
| 42900-42999 | 限流或配额错误 | 42901 模型调用配额不足 |
| 50000-50999 | 系统错误 | 50001 系统异常，50002 文件服务异常 |
| 51000-51999 | AI 服务错误 | 51001 模型调用失败，51002 模型返回格式异常 |

### 6.3 第三方模型 API 接口

系统通过模型适配层对接 ModelScope API、本地模型服务或旁路模型服务。

接口要求：

- 支持文本大模型调用。
- 支持多模态模型调用。
- 支持 Embedding 模型调用。
- 支持 Rerank 服务调用。
- 支持超时时间、重试次数和并发上限配置。
- 所有模型调用必须记录调用日志。
- 模型密钥不得在前端暴露。

模型服务不可用时，接口应返回明确错误码，业务层进入人工评分或重试流程。

### 6.4 文件存储接口

系统通过统一文件服务封装 MinIO 或本地文件系统。

接口要求：

- 支持文件上传。
- 支持文件下载。
- 支持文件预览。
- 支持文件删除或逻辑删除。
- 支持根据文件哈希判断重复文件。
- 支持生成内部访问地址。
- 支持文件访问权限校验。

知识库文件、学生提交文件和导出报告均通过统一文件接口管理。

### 6.5 幂等与并发控制

为避免重复提交、重复评分和并发覆盖，系统需要提供幂等与并发控制。

规则如下：

- 文件上传接口应根据 `task_id + student_id + file_hash + version` 判断重复文件。
- 学生连续点击提交时，前端按钮应置灰，后端应校验是否存在正在处理的同类提交。
- 教师保存评分时应携带数据版本号或更新时间，防止多人同时覆盖同一份评分。
- 批量任务应使用业务唯一键保证幂等，例如 `task_type + submission_id`。
- 失败任务重试时不得重复生成多份评分结果，应覆盖同一任务批次下的失败记录。
- 对状态已经进入 `PUBLISHED` 的提交，除成绩修正流程外，不允许普通评分接口再次覆盖。

### 6.6 文件存储策略

系统通过文件服务屏蔽 MinIO 和本地文件系统差异。

选型规则：

- MVP 单机部署优先使用本地文件系统，降低组件依赖。
- 多实例部署、文件量较大或需要对象存储能力时使用 MinIO。
- 两种方式应提供相同的上传、下载、预览和删除接口。

目录结构建议：

```text
/data/bisai/files/
  ├── submissions/{taskId}/{studentId}/{version}/
  ├── knowledge/{knowledgeBaseId}/
  ├── reports/{taskId}/
  └── temp/
```

清理与归档规则：

- 临时文件保留 24 小时后自动清理。
- 学生提交文件和评分报告默认长期保留，不自动物理删除。
- 超出版本保留数量的旧版本可归档，不直接删除。
- 删除操作优先采用逻辑删除，并记录操作日志。
- 归档数据应保证教师仍可按任务维度查询历史记录。

## 七、非功能需求

### 7.1 性能需求

- 普通页面响应时间应控制在 3 秒以内。
- 登录、查询任务、查看提交列表等普通接口响应时间应控制在 1 秒以内。
- 系统应支持 60 名学生在 10 分钟内完成成果上传。
- 单个文件大小 MVP 阶段默认不超过 200MB。
- 学生上传一个 50MB PDF 文件时，系统应在 30 秒内返回上传成功并进入待解析状态。
- 文档解析和智能评分采用异步任务处理。
- MVP 阶段至少支持 10 个并发解析任务。
- MVP 阶段至少支持 3 个并发大模型评分任务。
- 批量评价任务应显示执行进度。
- 班级统计报表导出应在 60 秒内完成。

### 7.2 安全需求

- 密码加密存储。
- 登录失败次数限制。
- 文件上传类型和大小校验。
- ZIP 文件安全解压。
- 模型输入防提示词注入。
- 模型输出进行 JSON Schema 校验。
- 教师和学生数据隔离。
- 关键操作记录审计日志。

### 7.3 可用性需求

- 系统应提供清晰的提交状态、解析状态和评分状态。
- 智能评价失败时应显示失败原因。
- 支持失败任务重试。
- 模型不可用时支持人工评分降级。
- 报表导出失败时应允许重新生成。

### 7.4 兼容性需求

- 支持主流浏览器访问。
- 支持 LoongArch 与麒麟服务器操作系统部署。
- 支持 ModelScope API 或其他模型服务替换。
- 支持 PostgreSQL + pgvector，后续可替换为其他向量数据库。
- 支持 Caffeine 本地缓存，条件允许时升级 Redis。

### 7.5 可维护性需求

- 模型服务、OCR 服务、Embedding 服务应通过适配器接入。
- 提示词模板应统一管理。
- 系统配置应支持后台维护。
- 模型调用日志应可追踪。
- 评分规则和评价模板应可配置。

### 7.6 数据备份与恢复需求

- 数据库应支持每日自动备份。
- 文件存储应支持定期备份，至少保留最近 7 天备份。
- 重要评分数据和提交文件不得物理删除，应优先采用逻辑删除或归档。
- 向量数据可通过文档切片和 Embedding 模型重新生成，但应记录模型版本。
- 数据库恢复时间目标 RTO 建议不超过 4 小时。
- 数据恢复点目标 RPO 建议不超过 24 小时。
- 管理员应能够查看最近备份状态。

### 7.7 部署需求

MVP 阶段推荐最小部署配置如下：

| 项目 | 最小要求 |
|---|---|
| CPU | LoongArch 服务器 CPU，4 核及以上 |
| 内存 | 16GB 及以上 |
| 磁盘 | 200GB 可用空间 |
| 操作系统 | 麒麟高级服务器操作系统 |
| JDK | 支持 LoongArch 的 OpenJDK 17 或以上 |
| 数据库 | PostgreSQL / openGauss / 达梦数据库 |
| 网络 | 能访问模型 API 时启用智能评分，不可访问时支持人工降级 |

部署方式：

- 业务系统、数据库和文件存储优先部署在 LoongArch + 麒麟服务器。
- 大模型服务可采用远程 API、旁路模型服务器或本地模型服务。
- MVP 阶段不强制要求 GPU。
- 如需本地部署多模态大模型，应单独评估 GPU/NPU、驱动和推理框架兼容性。

| 版本管理 | 学生重新提交后，系统生成新版本，默认以最新有效版本参与评分 |
| 文件解析 | 上传 Word、PDF、图片文件后，系统能生成解析文本或明确失败原因 |
| 智能核查 | 对包含缺失章节的报告，系统能输出缺失项、风险等级和证据说明 |
| 智能评分 | 系统能按评价指标返回建议分，超出满分的模型分数应被截断 |
| 教师复核 | 教师能修改单项分数、填写评语并发布最终成绩 |
| 通知消息 | 教师发布成绩后，学生端能收到站内消息提醒 |
| 报表导出 | 系统能导出学生个人 PDF 报告和班级 Excel 统计表 |
| 批量任务 | 教师批量启动评价后，页面能显示总数、成功数、失败数和执行中数量 |
| 安全控制 | 学生不能访问其他学生的提交文件和评价结果 |
| 降级能力 | 关闭模型 API 后，教师仍能通过人工评分完成成绩发布 |

## 十一、测试需求

### 11.1 功能测试

- 验证学生、教师、管理员核心业务流程是否完整。
- 验证任务创建、成果提交、智能评价、教师复核、成绩发布和报表导出是否可用。
- 验证异常流程，如误提交、成绩修正、解析失败和人工降级。

### 11.2 权限测试

- 验证学生不能查看其他学生提交文件和成绩。
- 验证教师只能管理自己课程下的任务和提交。
- 验证管理员可以访问系统配置和日志。

### 11.3 文件测试

- 验证 Word、PDF、图片、Excel、ZIP 文件上传和解析。
- 验证超大文件、非法扩展名、空文件和损坏文件处理。
- 验证 ZIP bomb、路径穿越和嵌套过深的防护。

### 11.4 模型异常测试

- 验证模型 API 超时、返回非法 JSON、返回分数越界、配额不足时的处理。
- 验证模型不可用时系统能进入人工评分模式。

### 11.5 性能测试

- 验证 60 名学生在 10 分钟内上传成果。
- 验证 10 个并发解析任务和 3 个并发评分任务。
- 验证 50MB PDF 上传 30 秒内返回成功。

### 11.6 国产化环境测试

- 验证 LoongArch + 麒麟系统下 JDK、数据库、文件存储和前端静态资源可运行。
- 验证模型 API 调用、OCR 服务、pgvector 或替代检索方案可用。

### 11.7 安全测试

- 验证登录失败限制、密码复杂度、接口鉴权和越权访问防护。
- 验证提示词注入文本不会覆盖系统评分规则。
- 验证模型输出内容经过格式校验和安全过滤。

## 十二、风险与约束

| 风险 | 影响 | 应对措施 |
|---|---|---|
| LoongArch 组件兼容性不足 | 部分依赖无法部署 | 优先使用 Java 技术栈和 API 化模型服务，准备人工降级方案 |
| ModelScope API 不可用 | 智能评分无法执行 | 支持旁路模型服务、本地模型服务或人工评分 |
| OCR 准确率不足 | 文档解析结果不完整 | 结合文档解析、多模态模型和教师人工复核 |
| 大模型评分不稳定 | 建议分存在波动 | 固定 Prompt 版本、评价模板和校准样本，教师最终确认 |
| 文件格式复杂 | 解析失败或预览失败 | 保留原文件下载和人工评分入口 |
| 批量任务过多 | 模型调用超时或限流 | 使用异步队列、并发限制和失败重试 |
| 知识库内容质量差 | RAG 评分依据不准确 | 知识库仅作为参考，教师可停用无效文档 |
| 网络不可用 | 远程 API 调用失败 | 保留本地基础功能和人工评分闭环 |
