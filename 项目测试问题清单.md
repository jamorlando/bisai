## 二、发现的问题

### BUG-AUTH-001 错误密码锁定逻辑触发空指针异常

严重级别：高  
影响用例：AUTH-04  
影响模块：登录认证、失败次数限制、账号锁定  
问题文件：`backend/src/main/java/com/bisai/service/CaptchaService.java`

问题位置：

```java
public boolean isLocked(String username) {
    LoginFailureEntry entry = failureStore.get(username);
    if (entry == null) return false;
    if (LocalDateTime.now().isAfter(entry.lockUntil)) {
        failureStore.remove(username);
        return false;
    }
    return true;
}
```

问题说明：

用户第一次输入错误密码后，`recordFailure()` 会向 `failureStore` 写入失败记录，但此时 `lockUntil` 为 `null`。下一次登录时，`isLocked()` 会执行：

```java
LocalDateTime.now().isAfter(entry.lockUntil)
```

由于 `entry.lockUntil` 为 `null`，触发 `NullPointerException`，最终接口返回 `500`。

实际结果：

```text
第一次错误登录：code=40101，提示用户名或密码错误
第二次错误登录：HTTP 500，message=操作失败，请稍后重试
```

预期结果：

```text
前 4 次错误登录：返回用户名或密码错误，并提示剩余次数
第 5 次错误登录：账号锁定 15 分钟
锁定期间登录：返回账号已被锁定，并提示剩余时间
```

影响范围：

- 任意用户只要输错一次密码，第二次登录检查就可能触发 500。
- AUTH-04 错误密码锁定功能无法正常验收。
- 如果真实用户输错密码，可能导致后续正确密码登录也异常，直到服务重启清空内存失败记录。
- 登录失败次数限制功能目前不可用。

建议修复：

在 `isLocked()` 中增加 `lockUntil == null` 判断。未达到锁定次数时，应视为未锁定。

建议修改为：

```java
public boolean isLocked(String username) {
    LoginFailureEntry entry = failureStore.get(username);
    if (entry == null || entry.lockUntil == null) {
        return false;
    }
    if (LocalDateTime.now().isAfter(entry.lockUntil)) {
        failureStore.remove(username);
        return false;
    }
    return true;
}
```

同时建议 `getLockRemainingMinutes()` 也增加空值保护：

```java
public long getLockRemainingMinutes(String username) {
    LoginFailureEntry entry = failureStore.get(username);
    if (entry == null || entry.lockUntil == null) {
        return 0;
    }
    long minutes = java.time.Duration.between(LocalDateTime.now(), entry.lockUntil).toMinutes();
    return Math.max(0, minutes);
}
```

### 6.2 发现的问题

#### BUG-TASK-001 任务发布时间未校验

严重级别：高  
影响用例：TASK-02  
问题文件：`backend/src/main/java/com/bisai/service/TaskService.java`  
问题位置：第 57-60 行 `createTask()`、第 69-79 行 `publishTask()`

实际结果：

```text
startTime=2026-12-31T23:59:59
endTime=2026-05-11T00:00:00
创建成功，发布成功
```

问题说明：`createTask()` 仅设置 `DRAFT` 后直接入库，`publishTask()` 只校验是否为草稿，未校验 `endTime` 是否晚于 `startTime`。

建议修复：在创建、更新、发布三个入口统一校验任务时间；截止时间早于或等于开始时间时返回明确业务错误。

#### BUG-FILE-001 关闭任务后仍允许学生提交

严重级别：高  
影响用例：TASK-05  
问题文件：`backend/src/main/java/com/bisai/service/SubmissionService.java`  
问题位置：第 156-263 行 `uploadFiles()`

实际结果：

```text
任务已关闭为 CLOSED
学生继续请求 /api/submissions/{taskId}/files
接口返回 success
```

问题说明：上传入口没有读取任务状态，也没有校验任务是否为 `PUBLISHED`、是否在提交时间范围内。

建议修复：上传前查询任务，只有 `PUBLISHED` 且当前时间处于允许提交范围内时才允许提交；`DRAFT`、`CLOSED` 均应拒绝。

#### BUG-FILE-002 非法扩展名上传失败后仍生成空提交记录

严重级别：中  
影响用例：FILE-02  
问题文件：`backend/src/main/java/com/bisai/service/SubmissionService.java`  
问题位置：第 190-210 行

实际结果：

```text
上传 exe/bat/js/html 返回“不支持的文件类型”
但 submission 记录数量从 0 增加到 1
```

问题说明：代码先创建 `Submission`，再逐个校验文件扩展名；发现非法扩展名时直接 `return Result.error(...)`，不会抛异常触发事务回滚。

建议修复：先完成所有文件的扩展名、大小、空文件、任务类型限制校验，再创建提交记录；或在错误时抛业务异常并回滚事务。

#### BUG-FILE-003 任务级 maxFileSize 未生效

严重级别：中  
影响用例：FILE-03 附加校验  
问题文件：`backend/src/main/java/com/bisai/service/SubmissionService.java`  
问题位置：第 202-238 行保存文件流程

实际结果：

```text
任务 maxFileSize=1 字节
上传 2 字节 PDF
接口返回 success
```

问题说明：系统级 200MB multipart 限制生效，但任务表中的 `maxFileSize` 没有在上传逻辑中被校验。

建议修复：上传前读取任务配置，逐个文件校验 `file.getSize() <= task.getMaxFileSize()`。

#### BUG-FILE-004 0 字节空文件可上传成功

严重级别：中  
影响用例：FILE-04  
问题文件：`backend/src/main/java/com/bisai/service/SubmissionService.java`  
问题位置：第 202-238 行保存文件流程

实际结果：

```text
上传 empty.pdf，文件大小 0 字节
接口返回 success，并生成文件记录
```

问题说明：上传逻辑没有校验 `MultipartFile.isEmpty()` 或 `file.getSize() == 0`。

建议修复：文件保存前拒绝空文件，并返回“文件为空，无法上传”之类的明确错误。

#### BUG-FILE-005 成绩发布后仍允许学生重复提交

严重级别：高  
影响用例：FILE-07  
问题文件：`backend/src/main/java/com/bisai/service/SubmissionService.java`  
问题位置：第 156-263 行 `uploadFiles()`

实际结果：

```text
教师发布学生提交成绩
学生再次上传 PNG
接口返回 success，并生成新版本
```

问题说明：上传逻辑只根据历史提交计算版本号，没有判断该学生该任务是否已有已发布成绩，也没有要求教师退回或重新开放后才能继续提交。

建议修复：上传前查询当前学生该任务最新提交或已发布成绩状态；存在 `PUBLISHED` 成绩时禁止提交，除非教师退回或任务重新开放。

#### BUG-FILE-006 ZIP 路径穿越未拦截或记录安全失败

严重级别：中  
影响用例：FILE-10  
问题文件：`backend/src/main/java/com/bisai/service/SubmissionService.java`  
问题位置：第 221-228 行文件保存流程

实际结果：

```text
traversal.zip 内包含 ../evil.txt
上传接口返回 success
未发现写出目标目录外文件，但也没有拒绝或记录安全失败
```

问题说明：当前上传阶段只把 ZIP 当作普通文件保存，不检查压缩包内路径。若后续解析 ZIP 时没有安全解压校验，存在 Zip Slip 风险。

建议修复：在 ZIP 解析或上传阶段校验 `ZipEntry` 规范化路径，发现 `../`、绝对路径或目标目录外路径时拒绝并记录安全事件。

#### BUG-FILE-007 任务级 allowedFileTypes 未生效

严重级别：中  
影响用例：FILE-01、FILE-02 附加校验  
问题文件：`backend/src/main/java/com/bisai/service/SubmissionService.java`  
问题位置：第 208-210 行

实际结果：

```text
任务 allowedFileTypes=PDF
学生上传 PNG
接口返回 success
```

问题说明：上传逻辑只校验全局 `ALLOWED_EXTENSIONS`，没有校验当前任务配置的 `allowedFileTypes`。

建议修复：上传前读取任务配置，将任务允许类型与实际扩展名对比；不在任务配置范围内的文件应拒绝。

### 7.2 发现的问题

#### BUG-SCORE-001 成绩发布后普通评分接口仍可覆盖成绩

严重级别：高  
影响用例：SCORE-06  
问题文件：`backend/src/main/java/com/bisai/service/ScoreService.java`  
问题位置：第 180-257 行 `saveTeacherScores()`

实际结果：

```text
教师发布成绩：scoreStatus=PUBLISHED，totalScore=85.00
再次调用 PUT /api/submissions/{id}/scores
接口返回 success
scoreStatus 被改回 TEACHER_CONFIRMED
totalScore 被覆盖为 20.00
```

问题说明：`saveTeacherScores()` 没有判断提交是否已发布。成绩发布后，普通评分保存仍可更新 `score_result`、`teacherComment`、`totalScore` 和 `scoreStatus`。

- 学生已查看的成绩可能被静默改写。
- 成绩修正记录无法完整反映发布后的分数变化历史。

建议修复：在 `saveTeacherScores()` 开头增加状态校验；若 `scoreStatus=PUBLISHED`，拒绝普通保存并提示必须走成绩修正接口。

#### BUG-SCORE-002 评分保存缺少并发冲突控制

严重级别：中  
影响用例：SCORE-08  
问题文件：`backend/src/main/java/com/bisai/service/ScoreService.java`  
问题位置：第 180-257 行 `saveTeacherScores()`

实际结果：

```text

窗口 B 基于旧页面再次保存
接口返回 success
最终记录为窗口 B 的评分和评语，没有任何冲突提示
```

问题说明：评分保存没有使用版本号、`updatedAt`、乐观锁或提交前状态校验。多窗口或多人协作评分时，后写会静默覆盖前写。

影响范围：

- 教师复核时可能丢失先前评分。
- 多教师或同一教师多标签页操作时没有数据变化提示。
- 发布前最后一次保存可能不是教师期望的结果。

建议修复：为提交记录或评分结果增加版本号/更新时间校验；前端提交保存时带上读取时的版本，后端发现已变化则返回冲突提示。

### 8.2 发现的问题

#### BUG-BATCH-001 异步任务重试后旧错误信息未清空

严重级别：中  
影响用例：BATCH-04  
问题文件：`backend/src/main/java/com/bisai/service/AsyncTaskService.java`  
问题位置：第 225-235 行 `retryFailedTask()`

实际结果：

```text
构造 FAILED 异步任务：error_message=manual failed
点击重试：status=PENDING，retry_count=0
任务最终执行 SUCCESS
但 error_message 仍为 manual failed
```

问题说明：代码中调用了 `task.setErrorMessage(null)`，但 `updateById()` 更新时空值没有写入数据库，导致成功任务仍残留旧失败原因。

影响范围：

- 前端展示任务详情时，成功任务可能仍显示旧错误。
- 运维排查异步任务时，状态与错误原因互相矛盾。
- 批量任务统计或日志分析可能误判成功任务为异常任务。

建议修复：对 `error_message` 字段使用允许空值更新的策略，或在重试时显式执行 `UPDATE async_task SET error_message = NULL`。

#### BUG-BATCH-002 批量任务缺少持久化的提交级去重

严重级别：中  
影响用例：BATCH-02、BATCH-03  
问题文件：`backend/src/main/java/com/bisai/service/TaskService.java`  
问题位置：第 95-191 行 `batchParse()`、`batchScore()`、`batchCheck()`

实际结果：

```text
同一批处理运行期间，activeJobs 可阻止重复启动
但批处理结束后，再次批量操作会重新给同一批提交入队
```

问题说明：当前只用内存 `activeJobs` 阻止同一 `taskId` 的当前批处理，未在数据库层按 `taskType + bizId` 检查已有任务。服务重启后 `activeJobs` 会丢失，批处理完成后也无法防止同一提交重复入队。

建议修复：创建异步任务前按 `taskType + bizId` 查询已有 `PENDING/RUNNING/RETRYING/SUCCESS` 任务；只为未处理、失败或用户明确选择重试的提交创建新任务。

#### BUG-BATCH-003 批量进度统计字段不区分任务类型

严重级别：低  
影响用例：BATCH-01、BATCH-02、BATCH-03  
问题文件：`backend/src/main/java/com/bisai/service/TaskService.java`  
问题位置：第 197-223 行 `getBatchProgress()`

实际结果：

```text
进度接口返回 total/parsed/scored/failed/running
failed 只统计 parseStatus=FAILED
核查失败、评分失败不会进入 failed 统计
```

问题说明：批量解析、批量核查、批量评分共用同一个进度接口，但失败统计只看解析状态，无法准确展示核查/评分批量任务的成功、失败、运行中数量。

建议修复：进度接口增加 `taskType` 参数或根据最近批处理类型统计；分别按 `parseStatus/checkStatus/scoreStatus` 返回 `success/failed/running/total`。

#### BUG-FE-001 前端构建失败，无法完成构建产物密钥扫描

严重级别：中  
影响用例：SEC-03  
影响命令：`cd frontend && npm run build`

实际结果：

```text
npm run build 失败
未生成 frontend/dist
无法扫描构建产物中是否包含 API Key
```

主要报错包括：

```text
src/api/knowledge.ts: post is declared but its value is never read
src/api/user.ts: del is declared but its value is never read
src/views/admin/Knowledge.vue: UploadRawFile | null | undefined 不可赋值给 File
src/views/teacher/ParseDetail.vue: "@/types" 没有导出 FileEntity
src/views/teacher/TaskEdit.vue: allowedFileTypes 类型不匹配
```

问题说明：源码扫描未发现 `ms-cb` API Key，但由于生产构建失败，`dist` 构建产物不存在，SEC-03 无法完整验收。

建议修复：先修复 TypeScript 构建错误，重新执行 `npm run build`，再扫描 `frontend/dist` 是否包含 `AI_API_KEY`、`ms-cb`、`api-key` 等敏感字符串。
